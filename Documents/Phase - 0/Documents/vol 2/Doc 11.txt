Document 11 — System Architecture & Repository Standards

Document ID: ARCH-001
Version: 1.0.0
Status: Approved

1. Purpose
This document defines the high-level system architecture, monorepo boundaries, package standards, and directory layouts of the AROH platform.

2. Monorepo Architecture
AROH uses an npm workspaces monorepo structure. This guarantees that shared utilities, database access layers, state stores, and UI design assets are authored once and consumed by all applications.

- apps/web: A Next.js App Router workspace containing the public homepage, authenticated user dashboard, Aros Wallet interface, CMS dashboard, and Admin/Operator panel.
- packages/ads: The Aroh Design System, containing color tokens, typography configuration, custom layout wrappers, buttons, input fields, and other presentation-only components.
- packages/asdk: The Aroh Software Development Kit, containing the centralized state store (Zustand), Firebase client helper library, Zod validation schemas, mock database fallbacks, and local emulator controls.

3. Data Flow
State management is unified and follows strict unidirectional flow:
- Authoritative Data Source: Firebase (Auth, Firestore, Storage) or local emulator fallback.
- Client State Authority: Zustand Store (`packages/asdk/src/store`).
- Presentation: React components (`apps/web` or `packages/ads`) consume Zustand selectors and dispatch actions.
- Actions: Zustand actions call the Firebase service layer, validate inputs with Zod, perform writes, and update the local state.

4. Repository Layout & Naming
- Component directories must contain PascalCase file names for the component and matching styles.
- Routing paths must use lowercase kebab-case.
- Shared utilities and services are named using camelCase (e.g., `firebaseClient.ts`).
