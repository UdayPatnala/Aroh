# AROH BOOTSTRAP DOCUMENTATION EXECUTION

You are responsible for generating the official AROH Bootstrap Documentation Suite.

This documentation will become the authoritative source of truth for every future implementation, documentation update, AI coding session, and engineering decision.

====================================================

OBJECTIVE

====================================================

Generate every bootstrap document one file at a time.

Each generated file must be complete, production-quality, internally consistent, and reference previous bootstrap documents where appropriate.

These bootstrap documents are intentionally concise.

They establish project intent, engineering rules, implementation scope, AI context, and development standards.

Detailed implementation documentation will be created later after implementation progresses.

Do not prematurely document APIs, database schemas, UI components, or implementation details.

====================================================

DOCUMENTATION STRATEGY

====================================================

The Bootstrap Suite contains only immutable project contracts.

Avoid unnecessary detail.

Focus on intent, standards, governance, philosophy, execution rules, and implementation guidance.

These documents should remain largely unchanged throughout Phase 1.

====================================================

DOCUMENT GENERATION ORDER

====================================================

Generate ONLY the current file.

Do NOT generate future files.

Wait for my approval after every file.

When I reply:

OK

Continue with the next file.

Repeat until the Bootstrap Suite is complete.

====================================================

OUTPUT FORMAT

====================================================

At the beginning of every response print exactly:

Generating:

<filename>

Then output ONLY the contents of that file.

No explanations.

No introductions.

No summaries.

No markdown outside the document itself.

No discussion.

====================================================

WRITING REQUIREMENTS

====================================================

Every document must:

Use professional engineering language.

Be implementation-oriented.

Avoid repetition.

Avoid duplication across documents.

Reference previous bootstrap documents instead of rewriting them.

Be future-proof.

Be AI-friendly.

Be human-readable.

Assume the reader is either:

• Senior Software Engineer

• Staff Engineer

• Principal Architect

• AI Coding Agent

====================================================

DECISION MAKING

====================================================

Make engineering decisions automatically.

Do not ask questions.

Prefer the objectively strongest solution.

Architecture is considered frozen.

Only change architecture if there is a measurable improvement in:

Security

Scalability

Maintainability

Performance

Developer Experience

Operational Simplicity

If architecture changes, document the reason.

====================================================

PHASE RULES

====================================================

Implement ONLY Phase 1 concepts.

Do NOT introduce Phase 2 or Phase 3 functionality.

You may prepare extension points.

You may not implement future functionality.

====================================================

QUALITY REQUIREMENTS

====================================================

Every document must satisfy:

Consistency

Maintainability

Scalability

AI readability

Human readability

No ambiguity

No contradictions

No duplicated information
